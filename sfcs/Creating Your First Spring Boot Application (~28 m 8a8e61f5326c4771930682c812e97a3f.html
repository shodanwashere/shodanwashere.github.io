1. Using Spring Boot to Create Applications

1.1. What is Spring Boot?

Before Spring Boot, it wasn’t uncommon for a developer to spend a few days wiring an MVC framework to a persistence framework and getting that talking to a database, all of this before any coding can actually take place. That’s way too much delay time.

Spring Boot takes an opinionated view on how a Spring application should be built. By doing so, you can now stand up a fully-wired up application from controller to database in just a matter of seconds.

Spring Boot makes building applications very quick, like Ruby-on-Rails, or Python.

Spring Boot got away from container dependence and really wants applications to be standalone Java apps.

1.4. Understanding Spring Boot Starters

Starters are a way to integreate a dependency on your project to get it all set up and ready to run by simply declaring it as a dependency. All Spring-based apps need to specify what pieces of the Spring Framework they’re going to utilized.

If you have a really complex app, you will also have a lot of dependencies, and those dependencies will have dependencies and so forth. This makes version mismatch and other headaches appear.

Spring Boot startes solve these headaches by allowing you to simply declare the started and it will take care of getting all the right dependencies and versions in place.

All dependencies in Maven-based Spring projects are declared in the pom.xml file:

  • Spring Boot Starter Test, for testing Spring Boot apps with testing libraries like JUnit;
  • Spring Boot Starter Web, the dependency we chose in Spring Initializr.

2.2. Connecting to the database

Now that we have both a base Spring Boot app and a Database set up, we need to connect the two so they can talk to each other.

We’ll first add a new started dependency to our app that will include Spring Data JPA, and then we’ll configure it to connect to the database. Let’s go to pom.xml and add a dependency to our dependencies. It’s another Spring Boot dependency whose artifactId is spring-boot-starter-data-jpa:

...
<dependency>
		<groupId>org.springframework.boot</groupId>
		<artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>
...

Another dependency we’ll need, because we’re using a Postgres database, is the Postgres JDBC Driver. Its groupId is org.postgresql and its artifactId is postgresql. There’s an additional property called scope. The scope of a dependency declares at which points of the applications lifecycle a dependency should be present. JDBC drivers’ scope is usually runtime. JDBC drivers are not required to compile our application, but they will be required at runtime for the database connection.

...
<dependency>
		<groupId>org.postgresql</groupId>
		<artifactId>postgresql</groupId>
		<scope>runtime</scope>
</dependency>
...

Now, all we need to do is set up the database connection for our Spring Boot application. These properties can usually be added/modified in the src/main/resources/application.properties file.

JPA Entities

Entities in JPA are just Plain Old Java Objects representing data that can be persisted in the database. An entity represents a table stored in a database. Every instance of an entity represents a row in the table.

Let’s take a look at our Session class. We can define it as an entity by using the @Entity annotation:

package com.shodan.conferencedemo.models;

import javax.persistence.Entity;

@Entity(name = "sessions")
public class Session {
}

The name property of the annotation must be the same as the name of the table in the database that this entity corresponds to. Now, let’s check the fields in that table and create the matching attributes in our class.

CREATE TABLE sessions
(
	session_id          SERIAL PRIMARY KEY,
	session_name        varchar(80)   NOT NULL,
	session_description varchar(1024) NOT NULL,
	session_length      integer       NOT NULL
);
@Entity(name="sessions")
public class Session {
    private Long    session_id;
    private String  session_name;
    private String  session_description;
    private Integer session_length;
}

Notice how, in our class I used, the exact same variable names as the column names in the table. By doing this, JPA will auto-bind to those columns, without needing to annotate them. If I wanted to use standard variable names, you would need to annotate them using the @Column annotation.

@Entity(name="sessions")
public class Session {
    // not annotating this one, you'll see why
		private Long id;

		@Column(name="session_name", length=80, nullable=false, unique=false)
    private String name;

		@Column(name="session_description", length=1024, nullable=false, unique=false)
    private String description;

		@Column(name="session_length", nullable=false, unique=false)
    private Integer length;
}

Two annotations you will have to add in any case are the @Id and GeneratedValue annotations. The first annotation denotes an attribute as being the table’s primary key, and the second denotes the way an ID is generated for a row. Both of these are always required.

@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long session_id;

// or

@Id
@Column(name="session_id")
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;

There are different types of value generation:

  • AUTO: used by default - JPA will chose the most adequate strategy in accordance with the database;
  • IDENTITY: JPA is informed that the values to be attributed to the unique ID are generated by an auto-increment constraint in the database.
  • SEQUENCE: JPA is informed that the generated values come from a Sequence in the database. If a sequence name is not specified, the default one will be used, global to all entities. If a Sequence name is specified, the provider will adopt this sequence to create primary keys for that table.
  • TABLE: a table is used to manage primary keys. This strategy is not recommended because of the overhead caused by reads and writes to the table.

2.4. Working with JPA Relationships

Now that we have two entity models in place, we can tie them together with a JPA relationship that will match the database table relationship.

Speakers and Sessions are connected via a UNION or JOIN table, so, we need to set up a many-to-many relationship in our models that will match that database relationship.

You need to pick one side of your relationship to be the owner or main definition point of the relationship. I’ll choose the Session entity, as sessions have speakers.

I’ll add a new attribute, a List of associated speakers, i.e.:

private List<Speaker> speakers;

There are annotations that denote relationships between entities. The relationship between Session and Speaker is many-to-many, so we’ll use the @ManyToMany annotation.

And because both entities are connected via a Join table, so we use the @JoinTable annotation to denote that. Inside, we’ll need to specify the name of the Join Table, the joinColumns, which are the columns belonging to the owning entity and the inverseJoinColumns, which are the columns belonging to the referring entity.

@ManyToMany
@JoinTable(
				name = "session_speakers",
				joinColumns = @JoinColumn(name="session_id"),
				inverseJoinColumns = @JoinColumn(name = "speaker_id"))
private List<Speaker> speakers;

CREATE TABLE session_speakers
(
    session_id integer NOT NULL REFERENCES sessions (session_id),
    speaker_id integer NOT NULL REFERENCES speakers (speaker_id)
);

Now we also need to define the relationship on the Speaker's, so we can make this relationship bidirectional.

We’ll create a list of sessions on the speaker, with getters, setters and a @ManyToMany annotation, but instead of having to define the entire relationship all over again, we can just specify which entity mapped the relationship:

@ManyToMany(mappedBy = "speakers")
private List<Session> sessions;

2.5. Working with Binary Data Types

Working with binary data can be a pain, so we need to handle a couple of “gotchas” to get things running correctly. Binary data comes in many different standards and formations, but at the end of the day, it’s all a bunch of bytes. Thankfully, Java has a primitive data type called byte that represents, well, a byte. We can use a byte array for any sort of binary data.

Let’s define the field for the speaker’s photo:

...
private byte[] speaker_photo;
...

Next we’ll specify some characteristics of this field through annotations. The first is @Lob, which stands for Large Object. Binary data can get extense, and this annotation notifies JPA that it is dealing with a field that usually stores large amounts of data.

Then there’s the @Type annotation, which we need to tell Hibernate (the JPA implementation we’re using) that the field holds binary data. It takes one parameter called type. To tell Hibernate this is a binary data field, we specify the type as org.hibernate.type.BinaryType.

...
@Lob
@Type(type="org.hibernate.type.BinaryType")
private byte[] speaker_photo;
...

Don’t forget to define getters and setters for it too.

Finally, we need to defined something in our application.properties:

...
spring.jpa.properties.hibernate.jdbc.lob.non_contextual_creation=true

This property is needed so the Postgres JDBC driver can create a Lob correctly on the application side.

2.7. Creating Spring API REST Controllers

Now we create controllers for our application, to deal with requests and process business logic. In our case, these controllers will be REST based. There’s a package for these too, com.shodan.conferencedemo.controllers.

I’m going to first create one for the Sessions, so SessionController:

package com.shodan.conferencedemo.controllers;

import com.shodan.conferencedemo.models.Session;
import com.shodan.conferencedemo.repositories.SessionRepository;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
@RequestMapping("/api/v1/sessions")
public class SessionsController {
		@Autowired
		private SessionRepository sessionRepository;

		@GetMapping
		public List<Session> list() {
				return sessionRepository.findAll();
		}

		@GetMapping
		@RequestMapping("{id}")
		public Session get(@PathVariable Long id){
				return sessionRepository.getOne(id);
		}

		@PostMapping
		@ResponseStatus(HttpStatus.CREATED)
		public Session create(@RequestBody final Session session){
				return sessionRepository.saveAndFlush(session);
		}

		@RequestMapping(value="{id}", method=RequestMethod.DELETE)
		public void delete(@PathVariable Long id) {
				sessionRepository.deleteById(id);
		}

		@RequestMapping(value="{id}", method=RequestMethod.PUT)
		public Session update(@PathVariable Long id, @RequestBody Session session) {
				// TODO: Add validation that all attributes are passed in, otherwise
				//       return 400 Bad Payload
				Session existingSession = sessionRepository.getOne(id);
				BeanUtils.copyProperties(session, existingSession, "session_id");
				return sessionRepository.saveAndFlush(existingSession);
		}
}

Let’s take a look at this Controller one component at a type and see what annotations and techniques were used.

  1. The class header:
    ...
@RestController
@RequestMapping("/api/v1/sessions")
public class SessionsController {
...

    We’re using two annotations here:

    • @RestController defines that this class is going to be a REST-based controller;
    • @RequestMapping tells Spring’s Web Application router what kind of path should be reserved for this controller. For example, when you interact with the path /api/v1/sessions, you’ll be interacting with the SessionsController and none other.
  1. The repository:
    ...
		@Autowired
		private SessionRepository sessionRepository;
...

    Spring will make use of autowiring to autowire a repository from the interface we created before. We will then be able to use all of the methods present in the JpaRepository we’re extending from.

  1. list() is a method that is supposed to return a list of all the Sessions stored in the database:
    ...
		@GetMapping
		public List<Session> list() {
				return sessionRepository.findAll();
		}
...

    Two things to note:

    • The @GetMapping annotation, which denotes that this method will be executed and its contents returned when the application receives a get request on the default mapped route (/api/v1/sessions);
    • The sessionRepository.findAll() method. The JpaRepository, when configured with an entity and its ID data type automatically configures itself to be able to query the database, and offers a variety of default methods with which to do so, despite being just an interface.
  1. The get() method, receives an ID and looks for a Session with that ID:
    ...
		@GetMapping
		@RequestMapping("{id}")
		public Session get(@PathVariable Long id){
				return sessionRepository.getOne(id);
		}
...

    Just like before it has a @GetMapping annotation, which means it will execute when a GET request is received, but it also has a @RequestMapping annotation, just like the class. The only difference is that now we’re defining a sort of sub-route for this method only. Notice how id is between curly braces. It’s set so that the method parameter can then be identified by the @PathVariable annotation. The repository has a method called getOne() that looks for one entity with a specific id.

  1. The create() method receives Session data through the request and then creates the Session in the database:
    ...
		@PostMapping
		@ResponseStatus(HttpStatus.CREATED)
		public Session create(@RequestBody final Session session){
				return sessionRepository.saveAndFlush(session);
		}
...

    Let’s look at this one closely:

    1. The @PostMapping annotation denotes that this method will execute when a POST request is made. There are not @RequestMapping annotations on this one, so it will execute on the default route.
    1. The @ResponseStatus annotation defines a custom success HTTP status code to be returned to the user. By default, Spring sends 200 OK on POST success. However, POST is usually used for creation of entities, and in those cases, its actually better to send a 201 Created code. That’s why the status code sent is HttpStatus.CREATED
    1. Notice how the parameter for the method takes the @RequestBody annotation. This means that the data pertaining to the mentioned parameter comes in the Request’s Body. The body will contain JSON data. When the web application routes to this method and sees that parameter, the JSON in the body will be automatically marshalled into a Session object and injected into the method.
  1. The delete() method deletes a Session with a specific ID:
    ...
		@RequestMapping(value="{id}", method=RequestMethod.DELETE)
		public void delete(@PathVariable Long id) {
				sessionRepository.deleteById(id);
		}
...

    This is very similar to how we performed our get() method, but we’re not using the @GetMapping annotation. We’re also not defining a @DeleteMapping annotation, and there’s a good reason for it: it doesn’t exist. Spring only has @GetMapping and @PostMapping annotations. To make use of other HTTP request methods, we need to specify them in the @RequestMapping annotation, as it can take a parameter called method that specifies what request method to use.

  1. The update() method updates a Session with a specific ID with new values for attributes:
    ...
		@RequestMapping(value="{id}", method=RequestMethod.PUT)
		public Session update(@PathVariable Long id, @RequestBody Session session) {
				// TODO: Add validation that all attributes are passed in, otherwise
				//       return 400 Bad Payload
				Session existingSession = sessionRepository.getOne(id);
				BeanUtils.copyProperties(session, existingSession, "session_id");
				return sessionRepository.saveAndFlush(existingSession);
		}
}

    We’ve done essentially the same as we did for delete(), but we are doing some other things.

    1. We’re using the copyProperties method from BeanUtils, which is a class with static convenience methods for Java Beans. In this case, we’re using a specific signature of copyProperties that takes 3 parameters: a source (the session object with updated attributes), the target (the existingSession in the database), and a string defining which properties should be ignored in the copy (”session_id”).
    1. We have a comment specifying that we should “Add validation that all attributes are passed in, otherwise return 400 Bad Payload”. Now this is actually good practice, and there are ways to do this. One way is to define a custom exception and give it a @ResponseStatus annotation, so that its Response Code is the one returned. When none is specified and the method fails, Spring only returns 500 Internal Server Error. Which is not nothing, but we prefer to know something about the error. For the purposes of this one segment, that is not the course, I will examplify:
      1. Let’s create a package called com.shodan.conferencedemo.exceptions, where we will store our custom exceptions.
      1. Then, let’s create our own exception, you can do that by creating a class and extending it off of the RuntimeException class:
        import org.springframework.web.bind.annotation.ResponseStatus;

@ResponseStatus(HttpStatus.NOT_FOUND)
public class BadUpdateException extends RuntimeException {
		public BadUpdateException(String exception){
				super(exception);
		}
}
      1. Then, in the controller class, import the exception by adding the following line to your imports:
        ...
import com.shodan.conferencedemo.exceptions.BadUpdateException;
...
      1. Finally, let’s write some code for that method to add that validation. If the attributes are not all defined, then an exception has to be raised:
        ...
		@RequestMapping(value="{id}", method=RequestMethod.PUT)
		public Session update(@PathVariable Long id, @RequestBody Session session) {
				{
						boolean b1 = session.name == null;
						boolean b2 = session.description == null;
						boolean b3 = session.length == null;
						if (b1 || b2 || b3) {
								throw new BadUpdateException();
						}
				}

				Session existingSession = sessionRepository.getOne(id);
				BeanUtils.copyProperties(session, existingSession, "session_id");
				return sessionRepository.saveAndFlush(existingSession);
		}
}

Let’s try sending a simple GET request to /api/v1/sessions. This is supposed to list all of the sessions stored in the database.

Ok… seems we might have a problem. Let’s check the terminal output to see what happened in our server.

Woah! That’s a big error. Let’s look in closer and try seeing what’s going on.

We have something referring “Infinite recursion”. This seems to me like a serialization problem, and you might have noticed why.

You know how our Session and Speaker entities have a many-to-many bidirectional relationship? Well, when Spring’s built-in JSON serializer gets data from the database and tries putting it into JSON, it writes to a session’s speaker list, and then each speaker in that list has their session list populated as well, so on and so forth. This is why the error refers “Infinite recursion”.

In a many to many relationship like this one, one way you can fix this is by adding annotations that the JSON serializer will use to ignore attributes in certain entities. We can use the @JsonIgnoreProperties annotation to ignore properties in certain attributes. Let’s for example ignore the speakers list inside of the sessions list of the Speaker entity, which can cause infinite recursion.

...
@ManyToMany(mappedby="speakers")
@JsonIgnoreProperties({"speakers"})
private List<Session> sessions;
...

And let’s do the same thing for the Session entity, but in the speakers list, ignore the internal sessions list instead:

...
@ManyToMany
@JoinTable(
				...
)
@JsonIgnoreProperties({"sessions"})
private List<Speaker> speakers;
...

Let’s try re-running the server and then sending that request again.

Aha! Much better now! Let’s keep testing our endpoints to see if we catch more of these serialization errors. Let’s try looking up /api/v1/sessions/1, where 1 is the id of a session.

Well here’s another serialization problem. Let’s see what’s causing this one.

So it seems that the JSON serializer is trying to serialize something called hibernateLazyInitializer in our Session entity. But we didn’t put anything like that in there. What gives?

We didn’t, but Hibernate did. When it identifies an entity, it creates some additional methods inside it for lazy and eager loading of data. We can’t have those serialized because they cause problems. We can use @JsonIgnoreProperties again.

...
@Entity(name="speakers")
@JsonIgnoreProperties({"hibernateLazyInitializer","handler"})
public class Speaker{
...

You might have noticed we’re also ignoring something called the handler. This is an extra field used by Hibernate for those lazy loading operations we’ve talked about.

Add that annotation to the class signature of both entities, then re-run the server and send the request again.

Seems fixed. Test the speaker endpoints to check this.

3. Working with Spring Boot Configuration and Environment Needs

3.1. How to Customize and Override Spring Boot

Spring Boot provides a lot of ways to override and customize your application. However, there is a specific order of precedence that Spring and Spring Boot use to properly set property values.

There are two places where configurations can be placed:

  • External Sources
    • Command line parameters
    • JNDI
    • OS environment variables
  • Internal Sources
    • Servlet parameters
    • Property files
    • Java configuration

Here is the order of precedence with which Spring Boot checks for application properties:

  1. Command line arguments
  1. SPRING_APPLICATION_JSON arguments
  1. Servlet parameters
  1. JNDI
  1. Java System Properties
  1. Environment Variables
  1. Profile properties
  1. Application properties
  1. @PropertySource annotations
  1. Default properties

Good practice is to pick one internal source, like property files, to set up application default properties, and then pick one external source to provide application property override values for specific environments.

For the purposes of this course, we’ll use the application.properties file as our internal source, and we’ll use environment variables for external configuration.

We’ve already looked at how to change the application.properties file, so we don’t need to go over that. For more properties to change, however, you can check https://docs.spring.io/spring-boot/docs/current/reference/html/common-application-properties.html.

3.2. Using Environment Variables as External Configuration

In our application.properties file, we can set certain properties to get their values from environment variables. Say the datasource URL for example. Instead of plopping the JDBC URL in our properties file, we can set it like this:

spring.datasource.url=${DB_URL}
...

Now, once Spring Boot starts, it will look for something called DB_URL. Following our order of precedence, it won’t be found anywhere above the environment variables. You can set one for your OS if you want, but you can also set your own for testing purposes in your IDE. For example, in IntelliJ IDEA:

  1. Go to “Run”>”Edit Configurations”
  1. Click on the little box in the “Environment Variables” field:
  1. To add a new Environment Variable, press the Cross in “User environment variables”:
  1. Give it a name and a value. For our example, the name would be DB_URL and the value is the JDBC URL.
  1. Press OK, then Apply and then exit the configuration edit menu.

To test that this configuration is indeed working, restart your application server and then send it a GET request. If a reply with all of the sessions/speakers was received, then your environment variable override is working.

3.3. How to Handle Different Environments

Setting different environments for different stages of a project’s lifetime is crucial. How can someone work with an app in development locally, then deploy to a nightly test server, staging and then into production?

Spring Boot uses a combination of Spring profiles and profile-specific application property files to accomplish this. Simply add any profile suffix to application-{profile}.properties and Spring Boot will use those properties instead of the ones in the default properties file.

Then, to use any of those profiles, you need to run your application with the -Dspring.profiles.active={profile} argument, and replace {profile} with your profile name. For example, if an application was started with -Dspring.profiles.active=dev, then Spring Boot would, on start up, look for a file named application-dev.properties.

You can use that argument when starting the application from CLI directly. For using an IDE, you can set the argument on your “Run”>”Edit Configurations” and then changing your “VM options” field:

3.5. Using Spring Properties in Code

You can set your own custom properties in your code, as long as they don’t used reserved property names. For instance, I could set up my own app.version property, and create a Controller that spits out that version when i visit the root directory /.

To examplify, I’ll do just that:

app:
	version: 1.0.0

Now, I’ll create a new controller on com.shodan.conferencedemo.controllers called HomeController:

package com.shodan.conferencedemo.controllers;

import java.util.Map;
import java.util.HashMap;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;
import org.springframework.web.bind.annotation.RequestMapping;

@RestController
public class HomeController{

		@GetMapping
		@RequestMapping("/")

				map.put("app-version", appVersion);
				return map;
		}
		
}

Let’s see how this works:

  1. First, we’re using the @Value annotation to inject the value of the app.version property into a private attribute. I covered this annotation in the following course:
  1. Our request handling method returns a Map. This is good for methods like the one I’m writing, because the JSON serializer is very good at taking Maps and converting them to JSON, as JSON is pretty much just a Map<String,String>.
  1. We put the app version we got from the property into the map and return it.

4. Deploying Spring Boot Applications

4.1. Spring Boot Containerless Architecture

If you were a Java developer back in 2005 and you wanted to deploy your application, you’d become very familiar with containers like Tomcat, Glassfish or Websphere.

These were all containers that complied with the JEE specification at that time, and when you deployed a Java application, a WAR file needed to be created so it could be installed in these containers.

Spring Boot apps are standalone: instead of installing your application into a container, Spring has embedded a container inside of the framework. The default container is Tomcat, but you can switch it for Jetty or Undertow if your prefer a different embedded container.

4.2. Switching out the Spring Boot Embedded Container

Switching out the default container is not difficult. But we need to review some facts first.

  • Normally, the Spring Boot Web Starter comes with Tomcat by default.
  • There’s no point in using two containers at the same time.

Now that we’ve got that settled, we can mess around with the dependencies to switch out the container. Let’s say we want to use Jetty instead of Tomcat. To do this, on our pom.xml we first need to tell Maven to exclude everything from the Tomcat Starter dependency when pulling the Web Starter.

...
<dependency>
		<groupId>org.springframework.boot</groupId>
		<artifactId>spring-boot-starter-web</groupId>
		<exclusions>
				<exclusion>
						<groupId>org.springframework.boot</groupId>
						<artifactId>spring-boot-starter-tomcat</groupId>
				</exclusion>
		</exclusions>
</dependency>	
...

And now, we add the Jetty Starter to the dependencies.

...
<dependency>
		<groupId>org.springframework.boot</groupId>
		<artifactId>spring-boot-starter-jetty</artifactId>
</dependency>
...

Have Maven sync up with your pom.xml changes and you’ve pretty much changed the container. Try it out for yourself and run your application again to see if it’s running on the new container.

4.3. Creating Executable JAR Deployments

There are two ways to create a basic standalone JAR file that can be used to run your application:

  1. Use Maven to create JAR files that can be run with the Java command.

    In your project directory, run the command

    $ mvn package

    Or, if you’re on IntelliJ (or some other IDE), press the “m” on the side to open the Maven tab:

    Then, select the window icon to “Execute Maven Goal”:

    And run the same command: mvn package

    If everything goes right, your application should package up with no problems:

    You can find your packaged JAR file in the project’s target folder.

    To run the JAR file, you need Java installed:

    $ java -jar conference-demo-0.0.1-SNAPSHOT.jar
  1. Use Maven to create a standalone executable JAR file.

    This can be done by going back to your pom.xml and adding a configuration tag to the Spring Boot Maven Plugin:

    <build>
		<plugins>
				<plugin>
						<groupId>org.springframework.boot</groupId>
						<artifactId>spring-boot-maven-plugin</artifactId>
						<configuration>
								<executable>true</executable>
						</configuration>
				</plugin>
		</plugins>
</build>

    Then, package the project the same way as in 1. The difference is that the JAR file produced in the target directory can now be run straight from the command-line, like so:

    $ ./conference-demo-0.0.1-SNAPSHOT.jar

The second method only works for UNIX or Linux-based systems. If you’re in Windows, this won’t generate an exe file to execute.

4.5. What About WAR Deployments?

To make a WAR deployment on a Maven project, we need to modify our pom.xml once again, and flag the embedded container as provided. This will tell Maven to not package Tomcat or any other container as the embedded container. Remove the Jetty dependency and the Tomcat exclusion from the Web Starter. Then add the Tomcat dependency but flag it as “Provided” in the scope:

...
</dependency>
<dependency>
		<groupId>org.springframework.boot</groupId>
		<artifactId>spring-boot-starter-tomcat</groupId>
		<scope>provided</scope>
</dependency>
<dependency>
...

Afterwards, we also want to tell Maven to package our app as a WAR instead of a JAR file:

...
<name>conference-demo</name>
<description>Demo project for Spring Boot</description>
<packaging>war</packaging>
...
