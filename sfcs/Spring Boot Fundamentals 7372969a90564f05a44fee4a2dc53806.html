2.3. Auto-configuration

Auto-configuration is a very useful and powerful feature of Spring Boot, which takes a convention-over-configuration approach.

Auto-configuration looks at the JARs on the class path and auto-configures beans automatically. It is designed to be non-invasive. For example, if you add your own beans, like a data source bean, the default embedded data-source backs off and Spring uses what you’ve provided. Auto-configuration is always applied after user-defined beans have been registered.

  • If you need to find out what auto-configuration is currently being applied and why, start your application with the --debug
  • An equivalent would be setting the logging.level.org.springframework property to DEBUG
  • You can also use the Spring Boot Actuator to achieve this, which will be talked about in later chapters.

2.4 Spring Boot Annotations

The @SpringBootApplication annotation is a very important annotation that has to be added to the main application class to indicate that the application should utilize Spring Boot. It is a shortcut annotation that applies 3 annotations in 1 statement:

  • @SpringBootConfiguration - Replaces @Configuration and annotates a class as a configuration
  • @EnableAutoConfiguration - Tells SpringBoot to configure beans
  • @ComponentScan - Tells Spring Boot to scan the current package and subpackages

3. Accessing Data with Spring Boot and H2

Now that the TrackZilla team has bootstrapped a simplea application, in this scenario, they have a need to persist data, for example, maintain a list of applications or create a list of tickets or a list of releases or tickets assigned to a release. Let’s talk a bit about this process of Data Persistence.

3.1. The H2 Database

  • It is an open-source database engine written in Java;
  • It is an in-memory database, which means that data is stored in memory and will not persist to the disk;
  • Good for Proofs of Concept and Dev environments for being a simple database
  • Can be administered via the H2 Console

It can be included in a project by adding its dependency in pom.xml:

...
<dependency>
		<groupId>com.h2database</groupId>
		<artifactId>h2</artifactId>
		<scope>runtime</scope>
</dependency>
...

Thanks to auto-configuration, since we are adding H2 as a dependency, Spring Boot knows that in this project, we will be connecting to the H2 database, so it auto configures H2 related properties like the database URL, username, password, etc.

The default properties are as follows:

spring:
		datasource:
				url: jdbc:h2:mem:testdb
				driverClassName: org.h2.Driver
				username: sa
				password:
		h2:
				console:
						enabled: false

What changes should be made for our demos?

spring:
		datasource:
				url: jdbc:h2:mem:bugtracker
				driverClassName: org.h2.Driver
				username: sa
				password:
		h2:
				console:
						enabled: true
						path: /h2

We’re using H2 because it’s a simple to configure, on-prem, on-machine data base that can be used for testing purposes. In a real production environment, you should use a proper database engine like Postgres, Oracle SQL, MongoDB, etc. You can also use one of those in this and the following modules if you so prefer.

3.2. ORM with JPA

Let’s talk about Object-Relational Mapping with the Java Persistence API.

There are several layers of abstraction in this process and I will list them in the order that they appear.

  1. First comes the Persistence Data Store. This is your database. It can be an H2 in-memory database or a server running MySQL, Postgres, MongoDB, etc.
  1. Above comes JDBC. It is a Java API to connect and execute queries against the database.
  1. JPA is an abstraction on top of JDBC that makes it easy to map Java objects to relational databases and vice-versa. It is just a specification or guidelines.
  1. JPA needs an implementation or instance provider. Spring uses Hibernate as an implementation.

Spring Boot selects a recent version of Spring Data modules for you. In the pom.xml, we’ve included the Data JPA starter. It comes with the following modules:

  • Hibernate, one of the most popular JPA implementations;
  • Spring Data JPA, which makes it easy to implement JPA-based repositories;
  • Spring ORM, which is the core-module that allows for Object-Relational Mapping in the Spring Framework.

3.3. Entities

Entities are objects that live in a database and they have the ability to be mapped to a database.

Entities are defined by the @Entity annotation.

Traditionally, JPA entity classes are specified in a persistence.xml file. However, with Spring Boot, this is not necessary. Instead, Enteties are scanned, just like Beans are. By default, all the packages below the main configuration class (@SpringBootApplication)

Let’s see an example. We’ll be making use of the Module 4 code from the GitLab repository. Let’s take a look at the pom.xml file:

...
<dependency>
		<groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <scope>runtime</scope>
</dependency>
<dependency>
		<groupId>org.springframework.boot</groupId>
		<artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>
...

We have two important dependencies: the H2 database, which is set to only be present at runtime, and the Data JPA Starter. So far so good.

Let’s look at Trackzilla’s project structure. It has 3 domain entities that represent the business data captured by the system. There are Applications, Releases and Tickets.

Let’s take a look at each of these 3 entities and see how they’re set up and how they’re related to each other.

First we’ll take a look at Application:

package com.pluralsight.entity;

import javax.persistence.*;

@Entity
public class Application {

    @Id
    @GeneratedValue(strategy = GenerationType.AUTO)
    @Column(name="application_id")
    private Integer id;

    @Column(name = "app_name", nullable = false)
    private String name;

    @Column(length = 2000)
    private String description;
    private String owner;

    public Application() {
    }

    public Application(String name, String owner,
                       String description) {
        this.name = name;
        this.owner = owner;
        this.description = description;
    }
    // ... getters and setters

    @Override
    public String toString() {
        return "Application{" +
                "id=" + id +
                ", name='" + name + '\'' +
                ", owner=" + owner +
                ", description='" + description + '\'' +
                '}';
    }
}

Application represents an Application that a bug or enhancement request or ticket will tracked against. It has an @Entity annotation on its class signature. If we wanted to use a different name for the table, we’d need a @Table annotation specifying that name. As it is not there, this entity will be mapped to a table called application.

Let’s look at the id property:

@Id
@GeneratedValue(strategy = GenerationType.AUTO)
@Column(name="application_id")
private Integer id;

It is annotated with the @Id annotation so that JPA will recognize it as the object’s ID. The @GeneratedValue annotation specifies that the value is generated automatically. The strategy type is set to AUTO which means JPA will talk to the database and automatically generate the ID in the way it thinks best.

Let’s look at two other properties in this entity:

@Column(name = "app_name", nullable = false)
private String name;

@Column(length = 2000)
private String description;

These two properties have been costumized with the @Column annotation. By default, a property is automatically mapped to a column in the table via its name, and its specifications are derived from the defaults set to their data type. However, these specifications - things like data length, the actual column name or whether that column can have no data at all or not - can be overriden using the @Column annotation. For example, the name property has the annotation denoting that it should be mapped to the app_name column and it sets nullable = false, meaning this column must always be filled on any row. The description property also has a @Column annotation, denoting that the description can hold up to 2000 characters.

3.3.1. Relationships

Let’s move on to the Ticket entity:

package com.pluralsight.entity;

import javax.persistence.*;

@Entity
public class Ticket {
    @Id
    @GeneratedValue(strategy = GenerationType.AUTO)
    private Integer id;

    private String title;
    private String description;

    @ManyToOne
    @JoinColumn(name = "application_id")
    private Application application;

    @ManyToOne
    @JoinTable(name ="ticket_release", joinColumns = @JoinColumn(name = "ticket_fk"), inverseJoinColumns = @JoinColumn(name = "release_fk"))
    private Release release;

    private String status;

    public Ticket() {
    }

    public Ticket(String title, String description,
                  Application application, Release release, String status) {
        this.title = title;
        this.description = description;
        this.application = application;
        this.release = release;
        this.status = status;
    }

		//... getters and setters
}

Ticket represents a submission of an enhancement request or bug tracking issue to the system for a given application. I see some interesting things here and I will talk to you about them: relationships.

If you’re not familiar with Entity Relationships, get out and go study. But if you are, you know there are three different types of relationships two entities can have with each other: One-To-One, One-To-Many and Many-To-Many.

How do we define these relationships with JPA? With annotations of course, and I will describe how they’re used:

  • @ManyToOne: One entity references another several times. For example:

    This is a many-to-one relationship because several employees can have the same department ID but every Department has an unique ID.

  • @OneToMany: One entity is referenced by another several times. This is essentially the other side of the @ManyToOne relationship, and they’re used in conjunction when making bidirectional relationships;
  • @OneToOne: One entity is referenced by one and only one entity.
  • @ManyToMany: One or more rows of one entity are referenced by one or more in another.

Normally, to define a relationship, we just define a property on an entity whose class is from another entity, and we use one of the relationship annotations. Let’s look at Ticket's application field:

@ManyToOne
@JoinColumn(name = "application_id")
private Application application;

As you can see, the @ManyToOne annotation is used, so this is a One-to-Many relationship. We’re giving the property the name application. If it was the same name used in the Application table for the entity ID, JPA would map it automatically. It isn’t, so we need to tell JPA how to map it.

For this case, since it is just a unidirectional relationship, we can just use the @JoinColumn annotation and define the name of the column that is being referenced on the target entity.

There’s another reference made in the release property:

@ManyToOne
@JoinTable(
				name ="ticket_release",
				joinColumns = @JoinColumn(name = "ticket_fk"),
				inverseJoinColumns = @JoinColumn(name = "release_fk")
)
private Release release;

There’s some interesting appendages here. Aside from being @ManyToOne, it’s using the @JoinTable annotation. That means that the relationship is not directly represented by having a reference inside of it, but stored in a new table, where each Ticket is related to a Release for being in the same row.

@JoinTable takes 3 attributes by default, the name of the Join Table, the joinColumns which are the columns in the Join Table that reference the source entity - for example, there’s the ticket_fk join column that references a Ticket; it can have a referencedColumnName but if it doesn’t, the name of the property tagged as the @Id is used instead -, and the inverseJoinColumns; same conceptas the joinColumns but they’re the columns in the Join Table that reference the entity on the other end of the relationship - for example, the release_fk join column references a Release and will be mapped to its ID.

Finally, a relationship annotation can have a few notable properties:

  • targetEntity: should be used if the data type of the attribute you’re setting is not exactly the same as the other entity class, and its value should be the class of the other entity, i.e., targetEntity = Release.class;
  • cascade: defines Cascade properties, i.e., what happens if a field that’s referenced another is deleted. There are diffent types of Cascade:
    • ALL - all operations are propagated from parent to child entity;
    • PERSIST - only propagates persistence operations (saving an entity) from parent to child entity;
    • MERGE - only propagates merge operations (copying the state of a given object onto a persistent object with the same ID) from parent to child entity;
    • DELETE - when a parent entity’s object is deleted, the referenced child row is also removed;
    • DETACHED - when a parent entity’s object is detached from the persistent context, then the child is also detached;
    • LOCK - if both parent and child entity objects are detached, when the parent is reattached to the persistent context, then the child is also reattached;
    • REFRESH - when a parent entity’s instance value is reread from the database, the child’s instance value is also reloaded;
    • REPLICATE - for when we have multiple data sources - when a parent entity instance is replicated from data source to another, the child entity instance is also replicated;
    • SAVE_UPDATE - when a parent entity’s instance value is saved and/or updated, the operation is propagated to the child entity instance.
  • fetchType: defines if a property related to a reference should be loaded when the object is loaded (Eager Loading), or only if it is requested (Lazy Loading). Lazy loading is especially good to use when dealing with object collections, as a list of referenced entities can be lazily loaded as you look for objects inside, instead of the entire list being loaded from the get-go - especially if the collection is large.

3.3.2. Repositories

The Spring Data Repository abstraction greatly reduces the need to have boilerplate code to implement data access for any kind of database. In this module, we have 3 repositories, each for one of the Entities. Let’s look at one of them, say the ApplicationRepository:

package com.pluralsight.repository;

import org.pluralsight.entity.Application;
import org.springframework.data.repository.CrudRepository;

public interface ApplicationRepository extends CrudRepository<Application, Long> {

}

The Spring Data repositories allow us to access data by creating JPA queries automatically from a method’s name. Notice how our repository is an Interface. In fact, all 3 of them are, but they have no implementation. That’s because Spring Boot automatically provides implementations for them.

Notice also how the repository extends from CrudRepository and this interface itself is typed. CrudRepository receives two types, the Class of the entity of the repository, and the type of the entity identifier - in this case, it is an Application repository, and the ID type is Long.

After setting that, Spring Data JPA sets up the rest for us.

Let’s execute this application to test this out. Let’s look at our entrypoint class:

package com.pluralsight;

// imports

@SpringBootApplication
public class FundamentalsApplication {

	private static final Logger log = LoggerFactory.getLogger(FundamentalsApplication.class);

	public static void main(String[] args) {
		SpringApplication.run(FundamentalsApplication.class, args);
	}

	@Bean
	public CommandLineRunner demo(ApplicationRepository repository) {
		return (args) -> {
			repository.save(new Application("Trackzilla","kesha.williams","Application for tracking bugs."));
			repository.save(new Application("Expenses","mary.jones","Application to track expense reports."));
			repository.save(new Application("Notifications","karen.kane","Application to send alerts and notifications to users."));

			for (Application application : repository.findAll()) {
				log.info("The application is: " + application.toString());
			}
		};
	}
}

As you can see, in the main class, we’re creating a CommandLineRunner Bean. A CommandLineRunner is essentially a small part of the program that will run when contained within a @SpringApplication, since a @SpringBootApplication is also a @SpringApplication, our CommandLineRunner Bean will always run. You can have more than one runner, but in that case, you also need to give them the @Order annotation, that denotes in which order the runners should run.

It is a Bean, so it will make use of dependency injection and inject an ApplicationRepository bean when it runs, meaning Spring Boot can manage everything. This runner will save 3 rows to the Applications table, pertaining to 3 different applications: “Trackzilla”, “Expenses” and “Notifications”. Then, it will obtain all of the applications and print out their data on the screen.

4. Configuring a Spring MVC Application with Spring Boot

Now that the TrackZilla team has persisted the data, the next step is to retrieve that data to display in a browser, essentially allowing the TrackZilla user base to view and interact with the data through a front-end system.

For starters, the dependency needed for Spring MVC is the Spring Boot Web Starter:

<dependency>
		<groupId>org.springframework.boot</groupId>
		<artifactId>spring-boot-starter-web</artifactId>
</dependency>

When this dependency is added to a project, Spring Boot’s autoconfiguration detects Spring MVC on the class path and provides all of the dependencies and autoconfiguration needed to develop web apps:

  • Dispatcher servlet
  • Error page
  • Servlet containter (Tomcat by Default)
  • WebJars

4.1. The Model-View-Controller Design Pattern

The MVC Design Pattern, which stands for Model-View-Controller is one of the most common design patterns used in software development. It decouples major components to promote faster development cycles and works well for developing large-scale web applications that need to be supported by large teams of developers.

4.1.1. The Model

The Model is the representation of data in a system, sometimes called an entity or domain model. It is typically data stored in a database and is programmatically represented as a Java object.

For example, in TrackZilla, there’s a screen that shows a list of applications. The model contains the list of applications itself.

4.1.2. The View

The View is responsible for displaying data, or the model. In Java, the view is typically done in JSP, JSF, etc. In our case study, we are using Thymeleaf which will be talked about at a later time.

4.1.3. The Controller

The Controller is responsible for directing incoming user requests to the correct resources and sending responses from those resources back to the user.

For example, the controller will get a request to see a list of TrackZilla tickets. Then it will direct that request to the ticket service, which returns the list of tickets, and the controller routes the list of tickets to the view for display.

Thymeleaf uses fragments, which allow you to define repeatable chunks of code that can be reused in other Thymeleaf templates. This works great for components which you wish to reuse across your web pages. For example, for a navigation bar.

4.2. Using MVC in a Spring Boot Project

Let’s open up the Module 5 project:

We’ve already talked thoroughly about the Model layer, so let’s talk about the Controller layer. We only have one controller that, unfortunate and conterintuitively, can be found in the com.pluralsight.web package, and is called TzaController:

package com.pluralsight.web;

// imports

@Controller
public class TzaController {
    private ApplicationService applicationService;
    private TicketService ticketService;
    private ReleaseService releaseService;

    @Autowired
    public void setApplicationService(ApplicationService applicationService) {
        this.applicationService = applicationService;
    }

    @Autowired
    public void setTicketService(TicketService ticketService) {
        this.ticketService = ticketService;
    }

    @Autowired
    public void setReleaseService(ReleaseService releaseService) {
        this.releaseService = releaseService;
    }

    @GetMapping("/applications")
    public String retrieveApplications(Model model){
        model.addAttribute("applications", applicationService.listApplications());
        return "applications";
    }

    @GetMapping("/tickets")
    public String retrieveTickets(Model model){
        model.addAttribute("tickets", ticketService.listTickets());
        return "tickets";
    }

    @GetMapping("/releases")
    public String retrieveReleases(Model model){
        model.addAttribute("releases", releaseService.listReleases());
        return "releases";
    }
}

This single controller is responsible for handling all incoming HTTP requests. As you can see, we have methods to handle requests like retrieveApplications() which is linked to the Service layerm as it contacts the applicationService to listApplications().

The service layer usually contains the business logic and retrieves the data using the repository layer, which we’re already familiar with, to send that data back to the view.

This class has the @Controller annotation which denotes it as a Spring MVC controller. Spring will consider this class when handling web requests.

Let’s take another look at the retrieveApplications() method:

@GetMapping("/applications")
public String retrieveApplications(Model model){
		model.addAttribute("applications", applicationService.listApplications());
		return "applications";
}

The @GetMapping annotation provides routing information. It denotes that any HTTP GET requests that have a particular path (root “/” by default if no path is specified) are mapped to that method.

How is Spring MVC configured? Typically, you’d need the @EnableWebMVC annotation to import the Spring MVC configuration, but Spring Boot imports it automatically when it sees the necessary dependencies on the classpath, and autoconfig kicks in.

Let’s talk about the View layer, which in this application is just a simple front-end using Thymeleaf, which is a templating engine that provides support for serving HTML. To use Thymeleaf, you need only two things: Thymeleaf itself, the templates and the fragments.

To get Thymeleaf, you can add it to your pom.xml dependencies:

<dependency>
		<groupId>org.springframework.boot</groupId>
		<artifactId>spring-boot-starter-thymeleaf</artifactId>
</dependency>

Once it is inserted, there’s really no extra configuration necessary, but if needed you can configure it in your application properties:

spring:

5. Building a RESTful Web Application with Spring Boot

Now that TrackZilla has a wide user base, the team has gotten requests from users to provide access to the TrackZilla data outside of the front-end, in API format. A big request is to have access to the list of tickets, along with their assigned release dates to display on custom interla facing reports used by executive leadership.

5.1. REST Architecture

REST stands for REpresentational State Transfer. It’s a set of guidelines that application developers use to design APIs. There are four principles that APIs follow:

  1. Data and functionality are considered resources and identify through a URI.
  1. Resources are manipulated using a fixed set of operations:
    1. GET retrieves a resource;
    1. POST creates a resource;
    1. PUT updates a resource;
    1. DELETE removes a resource;
  1. Resources can be represented in multiple formats, HTML, XML, Plaintext, JSON, among others.
  1. Communication between the client and endpoint is stateless.

RESTful APIs are easier to create and easier for many clients to consume when we compare them to something like SOAP.

5.2. REST on Spring Boot

Let’s take a look at the code for Module 6:

Let’s look at our TzaController:

package com.pluralsight.web;

import { ... };

@RestController
@RequestMapping("/tza")
public class TzaController {
    private ApplicationService applicationService;
    private TicketService ticketService;

    @Autowired
    public void setApplicationService(ApplicationService applicationService) { this.applicationService = applicationService; }

    @Autowired
    public void setTicketService(TicketService ticketService) {
        this.ticketService = ticketService;
    }

    @GetMapping("/tickets")
    public ResponseEntity<List<Ticket>> getAllTickets() {
        List<Ticket> list = ticketService.listTickets();
        return new ResponseEntity<List<Ticket>>(list, HttpStatus.OK);
    }

    @GetMapping("/applications")
    public ResponseEntity<List<Application>> getAllApplications() {
        List<Application> list = applicationService.listApplications();
        return new ResponseEntity<List<Application>>(list, HttpStatus.OK);
    }

    @GetMapping("/application/{id}")
    public ResponseEntity<Application> getApplication(@PathVariable("id") long id) {
        try {
            return new ResponseEntity<Application>(applicationService.findApplication(id), HttpStatus.OK);
        } catch (ApplicationNotFoundException exception) {
            throw new ResponseStatusException(HttpStatus.NOT_FOUND, "Application Not Found");
        }
    }
}

Some things here definitely look different. Let’s analyze.

First, you’ll notice the class is using a @RestController annotation instead of a simple @Controller. This simplifies creating RESTful services. This annotation is essentially a shortcut for two different annotations: @Controller and @ResponseBody.

Now, to note that a @RestController does not work with Thymeleaf, so methods cannot return models.

With this annotation here, Spring sets this controller up so that it now simply returns object data written directly to the HTTP Response as JSON. We’ve also set up a default route for this controller on /tza. This means that any method in this class must be contacted first on /tza. Say for instance that I wanted a list of all the applications. Then I need to contact /tza/applications. Simple enough.

Let’s look at two examples now of how we’re using request mapping for our methods.

@GetMapping("/applications")
public ResponseEntity<List<Application>> getAllApplications() {
		List<Application> list = applicationService.listApplications();
		return new ResponseEntity<List<Application>>(list, HttpStatus.OK);
}

@GetMapping("/application/{id}")
public ResponseEntity<Application> getApplication(@PathVariable("id") long id) {
		try {
				return new ResponseEntity<Application>(applicationService.findApplication(id), HttpStatus.OK);
		} catch (ApplicationNotFoundException exception) {
				throw new ResponseStatusException(HttpStatus.NOT_FOUND, "Application Not Found");
		}
}

In our getAllApplications() method, we’re giving it the route /applications. So, you’d only need to contact /tza/applications to invoke this method. OK so far.

On our getApplication() method, however, we’re setting a route with a dynamic value id. This can be setup by giving the respective method parameter the annotation @PathVariable and telling it what’s the name of the path variable that this parameter corresponds to. Thanks to this, we can now search applications by ID. Say for instance that I wanted to lookup the first application in the list. I could by invoking /tza/applications/0.

As I’ve mentioned before, because @RestController methods cannot return models because a @RestController is a @Controller with a @ResponseBody annotation, which denotes that the return value of a method is to be immediately serialized directly to the body of an HTTP Request.

That’s why we send a ResponseEntity, which represents the entire HTTP response. All we have to do is configure a ResponseEntity to be sent, which not only includes status code and headers, but also how to wrap the model we want to send in the response. Once the ResponseEntity is returned by the method, Spring takes care of the rest.

5.4. Response Formats

Let’s consolidate some understanding on how we can send a response. Let’s start with the ResponseEntity.

ResponseEntity is a generic type. As a result, we can use any type as the response type. For example:

return new ResponseEntity<List<Ticket>>(list, HttpStatus.OK);

This will return a response with a list of tickets serialized in JSON.

Since we specify the response status programmatically, we can return different status codes for different scenarios. There are some generally known status codes such as:

  • HttpStatus.OK - 200 (Everything is OK and the operation was executed successfully);
  • HttpStatus.BAD_REQUEST - 400 (The request was malformed or an error was found during its processing and the operation it describes will not be executed);
  • HttpStatus.CONFLICT - 409 (There is some conflict in either the request, or between it and the resource);
  • HttpStatus.NOT_FOUND - 404 (The requested resource could not be found).

You can find more status codes here: https://developer.mozilla.org/pt-BR/docs/Web/HTTP/Status.

5.5. Exception Handling

A combination of Spring and Spring Boot provides multiple options to customize responses for errors. For exception handling, we use the ResponseStatusException. It is a programmatic alternative to the @ResponseStatus annotation and is the base class for exceptions used to apply a status code to an HTTP Response.

We simply create an instance of it by providing an HttpStatus and, optionally, a reason and cause.

This approach provides more control over exception handling, as exceptions can be created programmatically.

Spring Boot provides a default error mapping, returning a JSON response with the HttpStatus and the exception message.

Let’s take a look at how this can be done. We’ll take another look at the getApplication method, used for looking up applications by ID:

6. Building a GraphQL Server with Spring Boot

The users of the TrackZilla application appreciate the REST API that was provided, but they now complain that the API returns way more data than they need. With GraphQL, we can leave it up to the users to determine the datapoints they’d like returned, offering extreme flexibility and efficiency.

6.1. An Overview of GraphQL

REST is a popular way to expose data from a server through an API. However, the repsonse returned from a REST API is very rigid and returns all of the data points as designed by the developer. GraphQL offers greater flexibility in the response returned than REST.

GraphQL is a query language for APIs or a syntax that describes how to ask for data, and is normally used to load data from a server to a clinet.

  • Allows the client to specify the exact data needed;
  • Allows aggregation of data from multiple backend sources.

Here’s a Sample Query and it’s response:


{
findAllApplications
		{
				id
		}
}

{
		"data":{
				"findAllApplications":[
						{ "id": "1" },
						{ "id": "2" }
				]
		}
}

You’ll need the Spring Boot GraphQL Starter, which automatically configures a GraphQL Servlet accessible at /graphql, a scanner for schemas in your class path and an endpoint for POST requests, and the GraphQL Java Tools library, which helps parse the GraphQL schema:

6.3. GraphQL Schemas

The GraphQL schema defines defines the data points offered via API. The schema contains the data types and the relationships between them, and the operations available: things like queries for retrieving data and mutations for creating, updating, reading and deleting data.

Behind the scenes, GraphQL schema files found on the class path are processed and the necessary beans are wired.

All schema files need the .graphqls extension and can be present anywhere. Let’s examplify this. Open the Module 7 project:

For the TrackZilla project, the scehmas are found in resources/graphql. Let’s take a look at our GraphQL schema, application.graphqls:

type Application {
  id: ID!
  name: String!
  owner: String!
  description: String!
}

type Query {
  findAllApplications: [Application]!
  countApplications: Long!
}

type Mutation {
  newApplication(name: String!, owner: String!, description: String!) : Application!
  deleteApplication(id:ID!) : Boolean
  updateApplicationOwner(newOwner: String!, id:ID!) : Application!
}

Notice there can only be one root Query and one Mutation type. Each file will contain all the query and all the mutation operations for the given type. Do you see it now? This schema applies only for the Application entity!

The first type, Application, defines the structure of the entity. Mostly like we have in the Java Class, but without annotations, constructors or getters and setters.

Then we have the Query type. This defines the query operations available for this type. You can see we have two: findAllApplications and countApplications.

Finally, the Mutations type. While Query defines operations that query the system state, Mutations are operations that make changes to the system’s state, like creating a newApplication, asking to deleteApplication or to updateApplicationOwner.

The schema denotes a lot of things, but that’s all it is and does: a scheme that denotes. There is no exacting specification and no instructions to perform any operations. As such, they have to be defined in code.

6.4. Query Operations

Let’s first implement a query resolver, which will use our repository to query the database. You can find ours at com.pluralsight.resolver.Query:

package com.pluralsight.resolver;

import com.coxautodev.graphql.tools.GraphQLQueryResolver;
import com.pluralsight.entity.Application;
import com.pluralsight.repository.ApplicationRepository;
import org.springframework.stereotype.Component;

@Component
public class Query implements GraphQLQueryResolver {
    private ApplicationRepository applicationRepository;

    public Query(ApplicationRepository applicationRepository) {
        this.applicationRepository = applicationRepository;
    }

    public Iterable<Application> findAllApplications() {
        return applicationRepository.findAll();
    }

    public long countApplications() {
        return applicationRepository.count();
    }
}

The Query Resolver allows Spring to automatically detect and call the right method in response to one of the GraphQL queries defined inside of the schema. Notice that the methods we’ve defined here have the same names as the ones defined in the schema. The query response is typically sent in JSON.

6.5. Mutation Operations

GraphQL also has the ability to update the data stored on the server by means of mutations, such as creating updating or deleting. This is done via a Mutation Resolver. They are defined in code by creating a class that implements the GraphQLMutationResolver Interface. The project has one at com.pluralsight.mutator.Mutation:

package com.pluralsight.mutator;

import com.coxautodev.graphql.tools.GraphQLMutationResolver;
import com.pluralsight.entity.Application;
import com.pluralsight.exception.ApplicationNotFoundException;
import com.pluralsight.repository.ApplicationRepository;
import org.springframework.stereotype.Component;

import java.util.Optional;

@Component
public class Mutation implements GraphQLMutationResolver {
    private ApplicationRepository applicationRepository;

    public Mutation(ApplicationRepository applicationRepository) {
        this.applicationRepository = applicationRepository;
    }

    public Application newApplication(String name, String owner,
                                      String description) {
        Application app = new Application(name, owner, description);
        applicationRepository.save(app);
        return app;
    }

    public boolean deleteApplication(Long id) {
        applicationRepository.deleteById(id);
        return true;
    }

    public Application updateApplicationOwner(String newOwner, Long id) {
        Optional<Application> optionalApplication = applicationRepository.findById(id);

        if(optionalApplication.isPresent()) {
            Application application = optionalApplication.get();
            application.setOwner(newOwner);
            applicationRepository.save(application);
            return application;
        } else {
            throw new ApplicationNotFoundException("Application Not Found", id);
        }
    }
}

The Mutation Resolver allows Spring to automatically detect and call the right method in response in one of the GraphQL mutations defined in the Schema. Just like with the Query Resolver, our mutations have the same names and parameters as was defined in the Schema.

6.5. Exception Handling

Adding error handling logic is straightforward. Similar to how we handled the errors in the RESTful API by raising an exception.

For GraphQL, we need an Exception class that implements the GraphQLError interface. This interface requires an extensions map typed to Map<String, Object> and 3 methods to be implemented: List<SourceLocation> getLocations, Map<String, Object> getExtensions and ErrorType getErrorType(). The extensions object serves to store extension information to the reported error. Here’s what was implemented in the TrackZilla project:

package com.pluralsight.exception;

// imports

public class ApplicationNotFoundException extends RuntimeException implements GraphQLError {

    private Map<String, Object> extensions = new HashMap<>();

    public ApplicationNotFoundException(String message, Long invalidApplicationId) {
        super(message);
        extensions.put("invalidApplicationId", invalidApplicationId);
    }

    @Override
    public List<SourceLocation> getLocations() {
        return null;
    }

    @Override
    public Map<String, Object> getExtensions() {
        return extensions;
    }

    @Override
    public ErrorType getErrorType() {
        return ErrorType.DataFetchingException;
    }
}

8.1. An Overview of Testing

In the course, the instructor says:

This is a bunch of bullshit and is honestly one of the worst quotes ever produced, so I will give another, as a fellow hacker, in response:

With this in mind, I’ll actually explain what Testing is and what it does. It is the process of evaluating and verifying that a piece of software or full application does what it is supposed to do. The benefits of testing include:

  • preventing bugs;
  • reducing development costs;
  • improving performance.

There’s different types of testing like Stress testing, where we test how much strain the system can take before it fails, Integration testing, where we ensure that software components or functions operate together, Regression testing where we check if new features break or degrade functionality, and much more, but in Software Development, we focus a lot on Unit Testing, in which we validate that each software unit performs as expected.

There’s usually libraries for these, and normally, their dependency is already included in the pom.xml file when you bootstrap an application: it is the Spring Boot Test Starter.

<dependency>
		<groupId>org.springframework.boot</groupId>
		<artifactId>spring-boot-starter-test</artifactId>
		<scope>test</scope>
</dependency>

This Starter includes libraries like JUnit, AssertJ, Hamcrest, and much more.

8.2. Unit Testing

Let’s look at unit testing REST APIs using JUnit, Mockito and Spring Test, also known as MockMVC. Like explained above, writing unit tests allows you to test individual units of code. JUnit is by far the most popular unit testing framework, so that’s what we’ll use.

Mockito is a great mocking framework, which we’ll use to provide data to our unit tests. Spring also has the @MockBean annotation, which works well with Mockito and allows to create configurable mock objects.

@WebMvcTest is an annotation used for controller layer unit testing. It’s often used to test one controller class at a time and works in conjunction with Mockito to mock the dependencies.

  • It scans only the controllers (@Controller and @RestController);
  • It does not load the full app context;
  • Dependent beans must be mocked.

Let’s open the final project, Module 9, to see these concepts in action:

Let’s open the TzaControllerUnitTest. You can find it at src/test/java/com.pluralsight.web.TzaControllerUnitTest:

// imports

@RunWith(SpringRunner.class)
@WebMvcTest(TzaController.class)
public class TzaControllerUnitTest {

Let’s examine this class:

...
@RunWith(SpringRunner.class)
@WebMvcTest(TzaController.class)
public class TzaControllerUnitTest {
		...

It’s using the @RunWith annotation, which denotes the runner class to be used to run test cases. A runner is a class created for JUnit that provides the functionality of the Spring TestContext Framework.

We also have the @WebMvcTest annotation, and we denot that the class being tested is TzaController. This will start the web application context but the embedded servlet container is not used, keeping testing lightweight.

@Autowired
private MockMvc mockMvc;

@MockBean
ApplicationService applicationService;

@MockBean
TicketService ticketService;

We have a MockMvc autowired bean. MockMvc is a powerful way to quickly test MVC controllers without having to start a full HTTP server. It also helps with processing HTTP responses by providing methods for “expectations”. I’ll explain that in a minute.

Notice we have two @MockBeans for the applicationService and the ticketService. These Beans will be created by Mockito and are able to be configured with specific mock behavior in a test. They would have to be included either way because they are dependencies of the TzaController.

Let’s look at our first unit test:

@Test
public void getAllApplications() throws Exception {
		mockMvc.perform(get("/tza/applications/"))
		.andExpect(status().isOk())
		.andExpect(content().contentType(MediaType.APPLICATION_JSON_UTF8))
		.andExpect(content().json("[]"));

		verify(applicationService, times(1)).listApplications();
}

First, we call mockMvc's perform method, which simulates an HTTP request to the controller. As you can see, it’s simulating a GET request to /tza/applications.

Then we use an andExpect method, which sets “expectations” for the response. Think of it as assertions. Whereas in a normal test, you perform your behavior and assert that you have a certain result, you simulate a request to the controller and expect the response to have specific characteristics, for example, to have returned 200 OK (.andExpect(status().isOk())), the content type to be JSON (.andExpect(content().contentType(MediaType.APPLICATION_JSON_UTF8))) and for it to have an empty list in the response (.andExpect(content().json("[]"));).

Finally, it uses the verify method to verify the number of times a mock method has been called. In this instance, we want to verify that the listApplications() method of our mock applicationService has been called exactly once.

The entire method is annotated with @Test to denote that it is, well, a test. When you want to run a test, or a battery of tests, Spring will detect it as a Test and run it, recovering and reporting its results.

8.3. Integration Tests

Integration testing allows for testing of the entire application and all of its layers and not just individual units, and it is useful to be able to do this without requiring deployment of the application.

The @SpringBootTest annotation is useful for integration testing and is chosen over @WebMvcTest because it starts the full application context, including the servlet, and does not costumize component scanning at all.

A @SpringBootTest looks for the main application class annotated with @SpringBootApplication, and uses that to start a Spring Application Context, simulating a client call.

Let’s look at our TzaControllerIntegrationTest:

package com.pluralsight.web;

// imports

@RunWith(SpringRunner.class)
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@AutoConfigureMockMvc
public class TzaControllerIntegrationTest {
    @LocalServerPort
    private int port;

    @Autowired

The class is using a Spring Runner and is set up with the @SpringBootTest annotation. Notice we have also specified the webEnvironment to use a random port every time it executes this test. This is great for testing, because it reduces the risk of port conflicts.

We’re also using the @AutoConfigureMockMvc annotation, which enables all auto-configuration related to MockMvc for testing.

@LocalServerPort

We have a @LocalServerPort annotation, which will inject the selected port onto this attribute. Since we defined we want a random port, this attribute will have a random value every time this test is run.

We also have a TestRestTemplate which allows to consume a REST API from a test case. Spring Boot already provides a TestRestTemplate automatically, so all you have to do is @Autowire that Bean to an object in the test class.

As you can see, we’re generating a ResponseEntity that expects to contain a List, corresponding to a response from a request sent by the restTemplate. It sends a GET request to http://localhost:port/tza/applications/ and, just like the ResponseEntity, also states what class of object it expects to receive.

Finally, we make an assertion regarding to the response's status code. We assert that it is equalTo(HttpStatus.OK), i.e., HTTP 200.
