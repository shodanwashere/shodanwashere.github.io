Architecture and Project Setup

Prerequisites:

  • Java JDK and JRE
  • Maven
  • An IDE (Instructor Choice is Intellij)
  • Tomcat

Sample App

Let’s start with a sample app, through which I’ll present some of the things we do with Java normally, but end up being a pain afterwards. I’ll be using IntelliJ IDEA and JDK 11 for this Sample App.

Open up IntelliJ (or some other IDE of your choice, but remember to follow the steps for your specific IDE) and create a new project.

We’ll name this app conference (1). Make sure your language is set to Java and that your Build system is set to “Maven”. Select your JDK version. For our demos, we’ll be using JDK 11 (2). Set your GroupId (3) and your ArtifactId (4). For naming’s sake, I gave it the same as the project name.

Click “Create” and your Java project will be created. The first file we’ll want to see is the Maven pom.xml file. This describes what Maven needs to know about how to build your project, what plugins to use, etc.

It should look something like this:

<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>

    <groupId>com.shodan</groupId>
    <artifactId>conference</artifactId>
    <version>1.0-SNAPSHOT</version>

    <properties>
        <maven.compiler.source>11</maven.compiler.source>
        <maven.compiler.target>11</maven.compiler.target>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
    </properties>

</project>

Don’t worry too much about it. You can see the groupId and artifactId you gave to the project. In the properties are specified what version of Java the source code was written for and what’s the target version to build for.

Let’s start by creating a few classes. For the following examples, I will be using com.shodan as my project’s base groupId. Here’s the structure for mine:

Model

Create a new package called com.shodan.model and inside it, a class named Speaker. It should have two attributes, a firstName, a lastName and for each, getter and setter methods:

package com.shodan.model;

public class Speaker {
    private String firstName;
    private String lastName;

    public String getFirstName() {
        return firstName;
    }

    public void setFirstName(String firstName) {
        this.firstName = firstName;
    }


    public String getLastName() {
        return lastName;
    }

    public void setLastName(String lastName) {
        this.lastName = lastName;
    }

}

Repository

Now we’ll create a Repository for our application (an interface and its implementation). Create a new package called com.shodan.repository. Using IntelliJ (as of now, I’m not sure if the same can be done in Eclipse or other IDE’s), I’ll first write the implementation, and from that class, extract an interface. I’ll name mine HibernateSpeakerRepositoryImpl:

package com.shodan.repository;

import com.shodan.model.Speaker;
import java.util.List;
import java.util.ArrayList;

public class HibernateSpeakerRepositoryImpl{
    public List<Speaker> findAll() {
        List<Speaker> speakers = new ArrayList<Speaker>();

        Speaker speaker = new Speaker();

        speaker.setFirstName("Nuno");
        speaker.setLastName("Dias");

        speakers.add(speaker);

        return speakers;
    }
}

If you wanna do the same thing I did, right click on your code and select “Refactor”>”Extract Interface”:

Give it a new name, for example, SpeakerRepository and select the one method, findAll, that we created in the implementation.

Once you click “Refactor”, an interface named SpeakerRepository will be created, and your Repository Implementation will be modified:

  • findAll will have an @Override annotation;
  • the class header will have an implements operator denoting that your class implements the SpeakerRepository interface.

Service

Create a new package named com.shodan.service and add a class to it named SpeakerServiceImpl. It should have one method named findAll, and make usage of the Repository to find all speakers:

package com.shodan.service;

import com.shodan.model.Speaker;
import com.shodan.repository.HibernateSpeakerRepositoryImpl;
import com.shodan.repository.SpeakerRepository;

import java.util.List;

public class SpeakerServiceImpl {

    private SpeakerRepository repository = new HibernateSpeakerRepositoryImpl();

    public List<Speaker> findAll() {
        return repository.findAll();
    }

}

Repeat the interface extraction process or write one yourself.

Application

Now, from your root java directory in your project, create an Application class: 

import com.shodan.service.SpeakerService;
import com.shodan.service.SpeakerServiceImpl;

public class Application {

    public static void main(String[] args) {
        SpeakerService service = new SpeakerServiceImpl();
        System.out.println(service.findAll().get(0).getFirstName());
    }

}

Once you run this code, it will read the first name of the first speaker in the list. My name is Nuno so…

Spring Configuration Using Java

For this chapter, we’ll walk through the configuration of a Spring application using just Java. We’re going to take that Sample application we created before and wire it up using Java annotations and the Java configuration loader.

For the purposes of this chapter, I will be using a duplicate of the sample application we’ve worked on before.

To begin configuring our application, we want to create a file where we can bootstrap everything at. Create a new class named AppConfig or something similar at your project root.

Now, to configure our Application, we’ll use the @Configuration annotation at class level to specify that this is a configuration class.

Inside our configuration file, we’ll define Beans.

Bean

In the previous course, it was defined that Spring is based on Inversion of Control, or Dependency Injection, as we usually call it. This means that objects declare their dependencies without having to fulfill them, because someone else will.

In Spring, the objects that form the backbone of your application and that are managed by the Spring [Dependency Injection] container are called beans. A bean is an object that is instantiated, assembled and otherwise managed by a Spring [Dependency Injection] container.

So Beans are essentially objects that make usage of Spring’s Dependency Injection. Let’s examplify this by declaring a bean for the SpeakerService. In our AppConfig, we can add a new Bean by defining a sort of “getter” method with a @Bean annotation:

import com.pluralsight.service.SpeakerService;
import com.pluralsight.service.SpeakerServiceImpl;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class AppConfig {

    @Bean(name="speakerService")
    public SpeakerService getSpeakerService() {
        return new SpeakerServiceImpl();
    }

}

Setter Injection

Setter Injection is a method of Dependency Injection via a “setter” method. Essentially, the dependent class will need:

  • a private attribute with the same class as the dependency class;
  • a setter method for that attribute;
  • changes made to the Bean in the Configuration file.

And the dependency class will need its own Bean in the Configuration file. This way, when Beans are being registered and created, via this Setter, we can immediately inject the dependencies needed. To examplify this, let’s take care of one of SpeakerServiceImpl's (the implementation class we defined in the configuration file) dependencies: it needs a SpeakerRepository.

First, in our AppConfig class, we’ll create a new Bean for the SpeakerRepository, using the implementation of it that we have, the HibernateSpeakerRepositoryImpl:

@Bean(name = "speakerRepository")
public SpeakerRepository getSpeakerRepository() {
		return new HibernateSpeakerRepositoryImpl();
}

Next, we’ll add the needed repository attribute and setter to the SpeakerService, or rather, its implementation.

In SpeakerServiceImpl, we modify and add the following:

private SpeakerRepository repository; // = new HibernateSpeakerRepositoryImpl();

public void setSpeakerRepository(SpeakerRepository repository) {
		this.repository = repository;
}

Because this is some method not in the interface, we’ll get yelled at by Java. Remember to add the method signature to the SpeakerService interface.

Finally, in our configuration file, we modify the SpeakerService Bean to inject a SpeakerRepository Bean into it:

@Bean(name = "speakerService")
public SpeakerService getSpeakerService(){
		SpeakerService speakerService = new SpeakerServiceImpl();
		speakerService.setSpeakerRepository(getSpeakerRepository());

		return speakerService;
}

Construction Injection

Construction Injection is just like Setter Injection, but, instead of using a setter for injecting dependencies, we do it straight on the Bean class’ constructor. For example, if we wanted to use Construction Injection, instead of defining the setter, SpeakerServiceImpl's constructor would look like this:

public SpeakerServiceImpl(SpeakerRepository speakerRepository){
		this.repository = speakerRepository;
}

Because this is a constructor, we don’t need to add it to the interface.

And the Bean would look something like this:

@Bean(name="speakerService")
public SpeakerService getSpeakerService(){
		SpeakerService speakerService = new SpeakerServiceImpl(getSpeakerRepository());
		return speakerService;
}

Linking the configuration file to the application

Finally, we can link the Configuration file to the application. Thankfully, it’s also quite easy.

Spring uses a special type of object called the ApplicationContext to save these configurations.

To make this link, we create an application context object using the implementation for annotated configurations (Java-only configuration) like so:

ApplicationContext applicationContext
		= new AnnotationConfigApplicationContext(AppConfig.class);

Now, when we want to get an instance of a class we’ve defined as a Bean, all we need to do is obtain it from the application context! How neat is that?

//SpeakerService service = new SpeakerServiceImpl();
SpeakerService service
		= applicationContext.getBean("speakerService", SpeakerService.class);

This works particularly well, because by default, Beans are defined as Singletons by Spring, so when you obtain a Bean, it’s only instanced once, and is never recreated again.

Using this implementation, you can test that the application still runs!

Spring Scopes and Autowiring

Now that we’ve seen some of the basic Spring configuration and how to create and implement beans inside of our application, let’s go a little deeper and look at Spring scopes and autowiring.

Scopes

There are five scopes available inside of Spring for us to configure a bean inside of our applications:

  • Valid in any configuration:
    • Singleton
    • Prototype
  • Valid only in web-aware Spring projects:
    • Request
    • Session
    • Global

All-Valid Configs:

Singleton

The Singleton design pattern restricts the instantiation of a class to a single object. It is the default bean inside of Spring. If you don’t give it a specific configuration, a Bean will be a singleton by default, which means there is one instance of it per Spring container or context.

The configuration for adding scope using Java configuration is quite simple. Let’s use our Sample Application to examplify and add scope to our SpeakerService Bean. Go to your AppConfig:

@Bean(name = "speakerService")
@Scope(value=BeanDefinition.SCOPE_SINGLETON)
public SpeakerService getSpeakerService(){
		return new SpeakerServiceImpl(getSpeakerRepository());
}

Prototype

The Prototype design pattern guarantees a unique instance per request, and thus, the scope inside of a Spring container mimics that design pattern: each time you request a Bean from the container, you are guaranteed a unique instance. It is essentially the opposite of a Singleton.

@Bean(name = "speakerService")
@Scope(value=BeanDefinition.SCOPE_PROTOTYPE)
public SpeakerService getSpeakerService(){
		return new SpeakerServiceImpl(getSpeakerRepository());
}

Web-Aware Configs:

The three other scopes that are available are:

  • The Request Scope, which returns a new Bean per HTTP request;
  • The Session Scope, which returns a single Bean per HTTP session, which stays afloat while the HTTP session is alive;
  • The GlobalSession Scope, which returns a single Bean per application.

I won’t go into much detail about these configurations, but there are other courses that explain this well.

Autowired

Autowiring is a great technique used to reduce the wiring up and configuration of code.

You can use the @Autowired annotation to autowire a bean to a method, if that method takes in an object of a class with a configured Bean. Let’s say for example, in our Setter Injection scenario.

We add the annotation to our repository setter:

@Autowired
public void setSpeakerRepository(SpeakerRepository speakerRepository){
		this.repository = speakerRepository;
}

Afterward, you can just use the no-arguments constructor to instance the SpeakerService. You are no longer required to invoke the setter. Because it is autowired, once the Bean is invoked, the setter will automatically run and inject the SpeakerRepository Bean into its parameters:

@Bean(name="speakerService")
@Scope(value=BeanDefinition.SCOPE_SINGLETON)
public SpeakerService getSpeakerService(){
		SpeakerService ss = new SpeakerServiceImpl();
		// ss.setSpeakerRepository(getSpeakerRepository());
		return ss;
}

Stereotype Annotations

To fully autowire our applications and not just single methods, we need stereotype annotations. There are several stereotype annotations we can use to define Beans while fully autowiring them:

  • @Component - just a normal Bean;
  • @Repository - denotes a class that is used as a repository object;
  • @Service - denotes a class where you put your business logic in.

To autowire our applications using the Java configuration, we simply need to add a component scan to our configuration file with an annotation that looks like this:

@ComponentScan({"com.shodan"})

The text inside specifies where Spring should begin looking for autowired and stereotype annotations.

Because we are fully autowiring the application, we no longer need any of these Bean definitions in the AppConfig. Even if I wanted to, the @Bean annotation is only declarable at a method level and not at a class level.

That being said, after deleting the methods and annotations of the Beans inside your AppConfig you can now go to the implementations of your Beans and add stereotype annotations to the class signatures. For example, to declare that SpeakerServiceImpl is a Bean (and to make it easier to look for stuff, that it is a Service), we add the @Service annotation:

@Service(name="speakerService")
public class SpeakerServiceImpl implements SpeakerService { ...

Remember, you need the annotation in the implementation class, not on the interface. Do the same on the repository like so:

@Repository(name="speakerRepository")
public class HibernateSpeakerRepositoryImpl implements SpeakerRepository { ...

In the class level, you can also declare a specific scope if you want using the @Scope annotation.

When performing autowiring, you can use whichever method of injection you want by placing the @Autowired annotation on either the setter or the constructor (but never both).

Spring Configuration Using XML

This chapter is optional. You’re not going to be performing XML Configuration on Spring in your lifetime, but if you want to read for reading’s sake, you can stick around and see what’s in store.

applicationContext.xml

The applicationContext file really is the root of an application configured with Spring. And just like we had in our Java configured application, the AppConfig is the same thing.

The name of the file doesn’t matter but applicationContext.xml is a standard. By default, Spring will look for a file named applicationContext.xml.

A simple view of Spring is that it’s just a HashMap of objects, and we define that Hashmap inside of our applicationContext.xml or AppConfig.java. The objects we have inside are just name-value pairs.

All of our XML configurations begin here.

There are some namespaces that the Spring developers have put together that help us in our configuration and validation of our files and make things a lot easier.

Let’s get started. Head to src/main/resources and create a file called applicationContext.xml.

With the file created, let’s first add the XML header and the namespaces. Namespaces are needed by an XML parser to understand what tags can be used and how they are used.

<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="
        http://www.springframework.org/schema/beans http://www.springframework.org/schema/beans/spring-beans.xsd">
</beans>

Now, to add a bean to our applicationContext.xml, we use the bean tag. It has some required properties, like name and class.

		<bean name="speakerRepository"
					class="com.shodan.repository.HibernateSpeakerRepositoryImpl"/>

This tag has to be placed inside the beans tag.

Setter Injection

To specify that we’re doing Setter Injection in XML Configuration, we add a property tag to the bean.

		<bean name="speakerService"
					class="com.shodan.service.SpeakerServiceImpl">
				<property name="speakerRepository" ref="speakerRepository" />
		</bean>

The property name has to be the same as the parameter name in your setter. The property reference needs to be the same as the name of the Bean we’re injecting.

Constructor Injection

In this case, constructor injection guarantees a contract for us. Our code and contract are defined when we create an instance because of the constructor. A disadvantage is that you need to define a constructor for each scenario you want to guarantee a contract for.

In XML Configuration, Construction Injection is index based and not name based as setter injection is.

To specify that we’re doing Constructor Injection in XML Configuration, we add a constructor-arg tag to the bean:

		<bean name="speakerService"
					class="com.shodan.service.SpeakerServiceImpl">
				<constructor-arg index="0" ref="speakerRepository" />
		</bean>

The constructor index needs to have the index of the constructor argument. Then you can either have a reference or a pair of type and value attributes where type is the classname of the value being passed to the argument, and value is the value being passed to the argument. If no type is specified, it will be java.lang.String by default.

Linking applicationContext.xml

We link it with the ApplicationContext interface, just like in the Java configuration, but we instead use the ClassPathXmlApplicationContext implementation, which extracts the configuration from an XML file. You can define this just like so:

ApplicationContext appContext
		= new ClassPathXmlApplicationContext("applicationContext.xml");

You can then use this ApplicationContext object just like in the Java Configuration to get beans and other resources.

Autowiring

In XML Configuration, there are four types of autowiring that you can do on a bean:

  • byType: Allows a property to be autowired if exactly one bean of that property type exists in a container (it will raise an exception if there are two beans with the same class but different names).
  • byName: Allows a property to be autowired if exactly one bean with the same property name exists in a container.
  • constructor: Allows autowiring into a constructor.
  • no: no autowiring

These are equivalent:

<bean name="speakerService"
			class="com.shodan.service.SpeakerServiceImpl">
		<constructor-arg index="0" ref="speakerRepository" />
</bean>
<bean name="speakerService"
			class="com.shodan.service.SpeakerServiceImpl"
			autowire="constructor"/>

It is the same thing as placing an @Autowired annotation on a constructor.

For Setter Injection, you can use autowire="byType" or autowire="byName".

Advanced Bean Configuration

Bean Lifecycle

InstantiationThe Bean is instantiated by Spring.
Populate PropertiesThe Bean’s properties are populated from a properties file or injected by other resources.
BeanNameAwareThe framework sets the Bean name and makes it aware of other resources.
BeanFactoryAware
Pre Initialization - BeanPostProcessors
InitializeBeanThe Bean is initialized utilizing properties that were just set.
initMethodAn initialization method that can be defined by the user.
PostInitialization - BeanPostProcessors

To create initMethods, you’ll need to add a dependency to your project, javax.annotation-api. You can add it to your project with the following XML:

<dependency>
		<groupId>javax.annotation</groupId>
		<artifactId>javax.annotation-api</artifactId>
		<version>1.3.2</version>
</dependency>

In any Spring configured Bean, you can add initMethods. There are two types:

  • PostConstruct methods always run after a bean’s constructor finishes running. The method can have any name you want, but you have to add the @PostConstruct annotation:
    @PostConstruct
public void initialize() {
		System.out.println("I run after being constructed!");
}
  • PreDestroy methods run before a bean is destroyed. The annotation is @PreDestroy:
    @PreDestroy
public void destruction(){
		System.out.println("OH NO THE HORROR");
}

FactoryBean

FactoryBean configuration is very similar to the init method configuration. It builds on the factory method design pattern and its functionality is built to work the same way.

It’s a great way to integrate legacy code into your application. You can enforce a contract in how your code is configured without creating a specific constructor for it.

Let’s examplify this. Based on our sample application, I’m going to create a new class called com.shodan.util.CalendarFactory, and I’m going to make it implement the FactoryBean interface. Since FactoryBean is typed, I’ll say its type is Calendar.

package com.shodan.util;

import org.springframework.beans.factory.FactoryBean;
import java.util.Calendar;

public class CalendarFactory implements FactoryBean<Calendar> {
		private Calendar instance = Calendar.getInstance();

		@Override
		public Calendar getObject() throws Exception {
				return instance;
		}

		@Override
		public Class<?> getObjectType() {
				return Calendar.class;
		}

		public void addDays(int num) {
				instance.add(Calendar.DAY_OF_YEAR, num);
		}
}

Now let’s move to our Java configuration at AppConfig. First, I’ll create the Bean for the CalendarFactory:

...

@Bean(name="cal")
public CalendarFactory calFactory(){
		CalendarFactory factory = new CalendarFactory();
		factory.addDays(2);
		return factory;
}

...

And now I’ll create a Bean for the Calendar itself:

...

@Bean(name="cal")
public CalendarFactory calFactory(){
		CalendarFactory factory = new CalendarFactory();
		factory.addDays(2);
		return factory;
}

@Bean
public Calendar cal() throws Exception {
		return calFactory().getObject();
}

...

Notice for a minute what we’re doing here. Through this factory we now have a Bean for the Calendar. And through that factory, we can modify how the Calendar is constructed before its Bean is initialized.

SpEL

The Spring Expression Language (SpEL) is a powerful expression language from the Spring Framework. It’s typically used in libraries but it can also be used to do some convenient changes to code.

You can use SpEL to:

  • Manipulate Objects;
  • Evaluate values at Runtime and change the behavior of code accordingly;
  • Evaluate and manipulate Application Configuration.

Let’s say we want to add a seed number to our Speaker model object.

private double seedNum;

public double getSeedNum() {
		return seedNum;
}

public void setSeedNum(double seedNum) {
		this.seedNum = seedNum;
}

Let’s say I want to define that seedNum at runtime. Let’s modify our Repository:

@Value("#{ T(java.lang.Math).random() * 100 }")
private double seedNum;

public List<Speaker>findAll(){
		...
		speaker.setSeedNum(seedNum);
		...
}

As you can see, we can dynamically inject values into our beans using the @Value and SpEL, even if its values are based on other dependencies that we don’t want to autowire. In fact, you can even use this method if you want to inject values based on methods or properties of other Beans.

Bean Profiles

Bean Profiles were implemented later in the Spring code base to help you adapt to environments.

They allow you to set up code that gets ran only in a specific environment which means you can swap configurations at runtime.

To add Bean Profiles, we just need to give our Bean a @Profile annotation and the name of the profile. There are no specific keywords, but principles should be followed as per usual. For example, if you want a Bean to only be used in dev environments, such as our mock HibernateSpeakerRepositoryImpl you could do something like this:

...
@Repository("speakerRepository")
@Profile("dev")
public class HibernateSpeakerRepositoryImpl implements SpeakerRepository {
		...

Then you can either set in your application’s configuration VM options…

… or in the command line …

… the option -Dspring.profiles.active=<profile_name>, where profile_name is the name of the environment you want to run on.
