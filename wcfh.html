Principles

The Three Core Principles for Clean Code

1) Clean code envolves using the right tool for the job.

2) High signal to noise ratio.

3) Self-documenting

“The main contributor to code-level documentation isn’t comments, but good programming style.”

Principle 1: The Right Tool for the Job

Hacking together a solution may seem creative and genial, but it often times is cumbersome, messy and contributes to dirty, messy code.

How do we select the right tool for the job?

Every Tech is Potentially Evil!

Principle 2: Maximize Signal to Noise Ratio

Code with a high signal to noise ratio is easier to understand.

Signal: Logic that follows the TED rule:

  • Terse - your code should not be excessively wordy
  • Expressive - it’s clear what the code is trying to do
  • Do one thing - it should have a clear responsibility and should do it well

Noise:

  • High cyclomatic complexity
  • Excessive indentation
  • Zombie code
  • Unnecessary comments
  • Poorly named structures
  • Huge classes
  • Long methods
  • Repetition
  • No whitespace
  • Overly verbose

Why is signal to noise ratio so important?

When reading code,

  1. Our brain is the compiler. You should follow the rule of 7 so as to not break the limit of human memory.
  1. The mess builds quietly, and subtly.

also known as “Don’t Repeat Yourself”

Principle 3: Self-documenting Code

Understanding the original programmer’s intent is the most difficult problem.

Well written code is self-documenting:

  • The intent of the code should be clear;
  • Layers of abstractions should be used;
  • Formatting for readability will

Naming

Dirty:

List<decimal> p = new List<decimal>() { 5.50m, 10.48m, 12.69m };
decimal t = 0;
foreach (var i in p){
	t += i;
}

return t;

Names should be carefully chosen so as to convey code intent more clearly, and avoid mathematical names like single letters.

The above example shows the kind of un-readability you can expect from using this kind of naming convention. You can derive variable name meanings from context, but that kind of mental gymnastics can be avoided through clean variable naming.

Clean:

List<decimal> prices = new List<decimal>() { 5.50m, 10.48m, 12.69m };
decimal total = 0;
foreach(var price in prices){
	total += price;
}
return total;

Naming Classes

Dirty:

  • WebsiteBO
  • Utility
  • Common
  • MyFunctions
  • JimmysObjects
  • *Manager / *Processor / *Info

Clean:

  • User
  • Account
  • QueryBuilder
  • ProductRepository

Guidelines:

  1. Use a noun (we want to create an instance of User)
  1. Specific name (creates a cohesive class: instance variables are used by many of the methods within that class)
  1. Name should lead the design toward Single Responsibility
  1. Avoid generic suffixes

  • Code doesn’t convey intent clearly - but encapsulating it in a function and giving it a good name can;
  • Task division - you want to accomplish a task, by dividing it into multiple atomic subtasks, that can be represented by functions (Single Responsibility).

Arrow Code

As our logic grows more complex, it’s common to see an arrow shape form. Arrow code is a sign that a method has high cyclomatic complexity.

Comprehension decreases beyon three levels of nested if blocks. - 1986 Study by Noam Chomsky and Gerald Weinberg

Solutions:

Extract Method

Before

		while
			do
			some
			complicated
			thing
		end while
	end if
end if

After

if

		doComplicatedThing()
	end if
end if

	while
		do some complicated thing
	end while

// CLEAN
private bool ValidUsername(string username){
	const int MinUsernameLength = 6;
	if(username.Length < MinUsernameLength) return false;

	const int MaxUsernameLength = 25;
	if(username.Length > MaxUsernameLength) return false;

	bool isAlphaNumeric = username.All(Char.IsLetterOrDigit);
	if(!isAlphaNumeric) return false;

	if(ContainsCurseWords(username)) return false;

	return IsUniqueUsername(username);
}

What is “too long”?

  • Whitespace & comments are used to separate lines of thought;
  • Scrolling is required;
  • Hard time coming up with a name for it;
  • Multiple conditionals;
  • Hard to digest.
Functions should rarely be over 20 lines, hardly ever over 100 lines and should have no more than 3 parameters. - Robert C. Martin, “Clean Code”
The maximum length (…) is inversely proportional to the complexity and indentation level of that function. So if you have a conceptually simple function that is just one long (but simple) case statement (…) it’s OK to have a longer function (…) if you have a complex function (…) adhere to limits all the more closely. - Linux style guide

Kinds of Exceptions

Unrecoverable

The most common:

  • Null reference;
  • File not found;
  • Access denied;
  • etc.

Recoverable

It’s not all lost:

  • Retry connection;
  • Try different file;
  • Wait and try again;
  • etc.

Ignorable

Truly ignorable exceptions are rare.

  • Logging click

You should never catch an exception that you can’t handle intelligently. Let it bubble up. If you have an error that you can’t swallow, that you can’t deal with on the spot, and that you can’t deal with higher up in the application, then your application is broken. The correct behavior for a broken application is to crash immediately. A broken application that tries to limp along in an inconsistent state is a danger to itself, its data and its users.

Classes

When to Create

Classes are like headings in a book.

You should create a class to:

  1. Model an object or new concept (either abstract or real-world);
  1. Refactor a class with low cohesion (split it into classes whose methods relate to the concept being modeled);
  1. Promote reuse (small, targeted code ⇒ reusage )
  1. Reduce complexity (solve once, hide away)
  1. Clarify parameters (can identify a group of data)

High Cohesion

  • Class responsibilities should be strongly-related:
    • Enhances readability;
    • Increases likelihood of reuse;
    • Avoids attracting the lazy.
  • Watch out for:
    • Methods that don’t interact with the rest of the class;
    • Fields that are only used by one method;
    • Classes that change often

Example:

Low Cohesion

  • Vehicle
    • Edit vehicle options
    • Update pricing
    • Schedule maintenance
    • Send maintenance reminder
    • Select financing
    • Calculate monthly payment

High Cohesion

  • Vehicle
    • Edit vehicle options
    • Update pricing
  • VehicleMaintenance
    • Schedule maintenance
    • Send maintenance reminder
  • VehicleFinance
    • Select financing
    • Calculate monthly payment

Sniffing out lack of cohesion

Classes with overly general and under-descriptive names attract magnet classes (classes that attract unrelated code due to poor naming conventions of what methods and data go where, leading to extremely bloated classes).

When is a class too small?

  1. Inappropriate intimacy (Rely heavily on one another and frequently call each other’s methods - a sign that the two classes should really be one class mixed together);
  1. Feature envy (One class that relies heavily on another - the two classes should be mixed together);
  1. Too many small moving parts (Can add unwanted complexity due to too many classes interacting with one another).

Comments

Comments lie. They’re not kept up to date. They’re either redundant or a crutch for bad code.

Are comments great, or a code smell? Both.

A Necessity and a Crutch

General rules:

  1. Prefer expressive code over comments;
  1. Use comments when code alone can’t be sufficient.

(Kill) Zombie Code

Zombies appear to be dead but are alive enough to ruin things.

Why zombie?

  • It doesn’t execute in runtime because it’s all commented;
  • It’s still present and often confused with real code, making maintenance infuriating.

There are 2 main reasons for the existence of zombie code:

  1. Risk Aversion (developers are scared of deleting code);
  1. Hoarding mentality (keeping code just in case they may need it later).

An example:

Why Zombie code hurts work

Ambiguity hinders debugging:

  • What did that section do?
  • Was this accidentally commented out?
  • Who did this

Ambiguity hinders refactoring:

  • Do I need to refactor this too?
  • How does my change impact this code?
  • What if someone uncomments it later?

  • Reduces readability
  • Creates ambiguity
  • Hinders refactoring
  • Adds noise to searches (by keyword)
  • Code isn’t “lost”

About to comment out code? Ask yourself:

  • When, if ever, would this be uncommented?
  • Can I just get it from source control later?
  • Is this incomplete work that should be worked via a branch?
  • Is this a feature that should be enabled/disabled via configuration?
  • Did I refactor out the need for this code?

Staying Clean

When to Refactor to Clean Code

  1. You need to work with the code
  1. Difficult to comprehend or change
  1. Test coverage to protect from regression

Accept No Broken Windows

Badly maintained code is often overlooked and mistreated, leading future maintenance workers that put their hands on it to treat it in the same way, or often worse: much like a building with broken windows, that will probably attract vandals in to break more windows and set fires.

Don’t accept broken windows. Take pride in your codebase and strive to make it better and cleaner.

Code Reviews

  1. Promotes proactive cleanliness
  1. Set guidelines

Work in pairs

  • Real-time code review
  • Increases quality
  • Naming and refactoring is easier

The Boy Scout Rule

Always leave the code you’re editing a little better than you found it. - Robert C. Martin
